Chunking :

	Parsing all of the .py and .md files iof the llm to make it a list of MinimalSource wich contains all informations contained by the LLM,
	under the form of structure of less than 2000 characters by default, and editable.

	For the python chunker, we parse all functions and classes, and if we encounter one larger than our max_chunk_size, we split it in smaller chunk

	For the md chunker, the core logic is the same but we parse by using titles (#, ##, ###) and we keep the same method of spliting in smaller chunks
	if it's larger than the largest amount we accept.

	They're both classes wich heritate from the abstract class BaseChunker, wich contain an abstract method of chunk sorting, and the function _split_large_text
	wich is the functiun for spliting in smaller chunks, wich is common to both the chunkers classes

BM25 :
	search and give a "score" based on the amount of time each words appear to give the toppic of the chunk, but do take care about the increase amount,
	as an exemple, if we go from 20times a words to 100, it will not take it as such an increase, 'cause it already determine we were speaking about that, and will
	go look for new words occurences, or thing more relatable not counting only on occurences and size of the chunk.

CLI Commands:

	index:
		Will parse and ingest all relevant files (.py, .md) from the target repository directory (vLLM).
		Delegates text processing to the chunkers to respect the max_chunk_size limit.
		Converts extracted chunks into a searchable BM25 inverted index.
		Saves both the statistical index format and the mapped MinimalSource records locally to the disk for fast retrieval.

	search:
		Will perform a real-time semantic query against the previously generated BM25 index.
		Reads the compiled dataset directly from the disk.
		Iterates and ranks the indexed documents to return the top `k` most relevant matches (scoring).
		Outputs the matching raw text snippets and their structural mapped location (file path, starting/ending character index).

	search_dataset:
		Will load a JSON dataset containing questions (like UnansweredQuestions) and iterate over them.
		Must use `tqdm` to display a progress bar.
		For each question, it will query the BM25 index to find the top `k` sources (MinimalSource).
		It will wrap these results and save them back into a new JSON file matching the Pydantic `StudentSearchResults` model.

	answer [todo]:
		Will execute the BM25 retrieval for a single text query to gather relevant chunks.
		Must combine the user's query and the retrieved text contexts to form a prompt.
		Will use a Local LLM (Qwen/Qwen3-0.6B) through a library like transformers/vLLM to generate a natural language answer based exclusively on the provided context.

	answer_dataset [todo]:
		Must read the output generated by `search_dataset` (which contains questions + their top-k retrieved sources).
		Will iterate over every question (using a `tqdm` progress bar).
		For each item, it sends the context and the question to the Qwen LLM to generate the answer.
		Finally, outputs a JSON file validated against the `StudentSearchResultsAndAnswer` Pydantic model.

	evaluate [todo]:
		Will take two JSON files: the results predicted by our system and the "Ground Truth" provided in AnsweredQuestions.
		Must compute the Recall@k metric for each question.
		A source is valid (found) if there is at least a 5% character overlap between a retrieved chunk and a ground truth chunk.
		Will aggregate these scores and output the system's global performance metrics (Recall@1, @3, @5, @10).
